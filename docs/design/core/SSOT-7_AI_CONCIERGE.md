**目次**

1\. プロジェクト概要

1.1 プロダクトビジョン

客室TVを「静的なメニュー画面」から「ホテル体験の中心軸」へ変革する。AIコンシェルジュを核とし、ゲストの滞在体験全体を自然言語でサポートする客室TVシステムを構築する。

1.2 ターゲット

- 対象ホテル規模: 中小規模ホテル・旅館 (10～100室)

- 初期ターゲット: 100店舗規模での展開

- ハードウェア: Google TV / Chromecast with Google TV ベース

- エンドユーザー: ホテルゲスト（国内外）+ ホテルスタッフ（管理側）

1.3 競合優位性

既存のホテルTVソリューション（アルメックス等）は「静的メニュー画面 +
VOD」で止まっている。本システムは以下の点で差別化する:

- AIコンシェルジュによる自然言語インタラクション

- Ambient Intelligence UIによるコンテキスト応答型情報提供

- ホテルスタッフが自然言語でUIをカスタマイズ可能

- PMS連携によるパーソナライズ体験

- 配信プラスHub連携による収益化

2\. システムアーキテクチャ

2.1 全体構成図

システムは4つの主要レイヤーで構成される:

  ----------------------------------------------------------------------------

  デバイス層              Google                       Capacitor (Android TV)
                          TV端末のネイティブアプリ     Nuxt/Vue資産をNative化

  アプリケーション層      UIレンダリング・状態管理     Vue/Nuxt + WebSocket

  AI層                    コンシェルジュ応答・UI生成   Claude API +
                                                       Whisper/Google STT

  バックエンド層          テナント設定・PMS連携        hotel-saas API +
                                                       PostgreSQL
  ----------------------------------------------------------------------------

2.2 デバイス構成

Google TV / Chromecast with Google TV
をベースデバイスとして使用する。Kiosk
Modeでカスタムランチャーを全画面表示し、Google
TV標準のホーム画面（広告・おすすめ表示）を置き換える。

**デバイス管理:** Android Enterprise / Google Endpoint
ManagementによるZero-Touch
Enrollment。箱を開けたら自動でKiosk設定が完了する。

**セキュリティ:** USB無効、ADB無効、設定画面アクセス不可、Factory Reset
Protection有効。

2.3 テンプレートエンジン

テンプレートベースのデザインシステムを採用する。Builder.ioのような外部ビジュアルエディタは100店舗規模ではコストが爆発するため、自前のテーマ設定システムを構築する。

3層テンプレートアーキテクチャ

  ---------------------------------------------------------------------------------------
  **層**            **管理者**        **内容**                       **編集方法**
  ----------------- ----------------- ------------------------------ --------------------
  Layer 1:          IYASAKA           5-10種類のデザインプリセット   Figma AI + MCP →
  テンプレート                                                       Vueコンポーネント

  Layer 2:          ホテルスタッフ    色・ロゴ・フォント・画像       自然言語
  テーマ設定                                                         AIアシスタント

  Layer 3:          ホテルスタッフ    施設情報・メニュー・お知らせ   管理画面フォーム +
  コンテンツ                                                         tiptap
  ---------------------------------------------------------------------------------------

2.4 コスト比較（100店舗）

  --------------------------------------------------------------------------------
  **方式**          **月額**          **開発工数**      **レイアウト破壊リスク**
  ----------------- ----------------- ----------------- --------------------------
  Builder.io        \$5,000+/月       低                低

  GrapesJS          約\$75/月         中～高            中（要制限設計）

  テーマJSON +      約\$75/月         低～中            ゼロ
  tiptap（推奨）                                        
  --------------------------------------------------------------------------------

推奨理由:
GrapesJSのようなフルビジュアルエディタは「自由度が高すぎる」リスクがある。ホテルスタッフがレイアウトを破壊する事故が頻発するため、フォームベースのテーマJSON方式が最適。

3\. UIデザイン

3.1 コンセプト: Ambient Intelligence UI

従来の「選んで→押す」のリモコンナビゲーションから脱却し、TVを「反応する空間」として再定義する。AIコンシェルジュが核であるため、UIの主役は「対話そのもの」となる。

3つのUI層

  --------------------------------------------------------------------------------------------------------------------------------
  **層**                  **役割**                                                                         **モード**
  ----------------------- -------------------------------------------------------------------------------- -----------------------
  Layer 1: アンビエント層 時間帯・天気・予約状況に応じて自動変化。ゲストが何もしなくても価値がある。       受動的

  Layer 2: 対話層         AIコンシェルジュとの会話がメインUI。画面は会話内容に応じて動的にモーフィング。   インタラクティブ

  Layer 3: 探索層         従来型メニューも残すがサブ的位置づけ。空間的ナビゲーションで再発明。             能動的
  --------------------------------------------------------------------------------------------------------------------------------

3.2 TV固有の設計原則

スクロール禁止の原則

TV画面はスクロールなしを鉄則とする。リモコンのD-pad操作では「どこまで続くか分からない不安」と「行きすぎて戻る苦痛」が生まれるため。

  -----------------------------------------------------------------------
  **制約**                            **値**
  ----------------------------------- -----------------------------------
  操作方法                            ↑↓←→ + OK の5ボタンが基本

  視聴距離                            2-3m離れている

  画面サイズ                          16:9固定（32-55インチ）

  フォントサイズ                      最小24px（シニアモードは32px）
  -----------------------------------------------------------------------

スクロールの代替手法

情報が多い場合はスクロールではなく以下の4つの手法で解決する:

1.  ページネーション: ←→でページ切替（例: 周辺レストラン一覧 1/3ページ）

2.  タブ切替: ↑↓でカテゴリ切替（例: 朝食/昼食/夕食/ルームサービス）

3.  フォーカス展開: 選択すると詳細がインライン展開（例: 温泉一覧）

4.  部屋遷移: 画面全体の遷移（「戻る」ボタンで1アクションで帰れる）

デバイス別UI戦略

  ------------------------------------------------------------------------------
  **項目**          **TV（リモコン）**    **タブレット**       **スマホ**
  ----------------- --------------------- -------------------- -----------------
  スクロール        禁止                  セクションスナップ   フリー

  ナビゲーション    D-padフォーカス移動   タッチ               タッチ

  レイアウト        ゾーンベース          カードセクション     シングルカラム

  情報超過時        ページネーション      スクロール           スクロール

  AI会話表示        最新1ターンのみ       直近3ターン          全履歴

  フォント最小      24px                  16px                 14px
  ------------------------------------------------------------------------------

4\. アンビエント層設計

4.1 時間帯別状態遷移

TVを「待機中」にしない。常にコンテキストに応じた情報が生きた状態で流れている。

  -------------------------------------------------------------------------------------------------------------
  **時間帯**        **背景**                         **表示情報**                             **雰囲気**
  ----------------- -------------------------------- ---------------------------------------- -----------------
  朝 (6:00-10:00)   ホテル周辺の朝の風景映像         挨拶・天気・朝食状況・おすすめ           calm

  昼 (10:00-17:00)  周辺観光スポットスライドショー   COリマインダー・地域イベント・おすすめ   active

  夕方              夜景 or 館内雰囲気映像           夕食・バー・費切風呂空き状況             warm
  (17:00-22:00)                                                                               

  深夜 (22:00-6:00) 時計だけ薄く表示 or 画面OFF      なし（邪魔しない）                       最小限

4.2 PMS連携によるパーソナライズ

PMSからの予約データ連携により、ゲスト名・宿泊プラン・食事時間・アレルギー情報まで反映し、体験をパーソナライズする。

- チェックイン時:
  ウェルカム画面にゲスト名表示、Wi-Fi自動接続案内、食事時間表示

- 滞在中: 時間帯に応じたプッシュ通知（夕食準備完了等）

- チェックアウト前: 忘れ物チェックリスト、レビューQR、次回予約割引QR

5\. AIコンシェルジュ設計

5.1 対話層のUI動作

ゲストが話しかけると、画面が会話内容に応じてモーフィングする。AIの応答にメタデータを付与し、「何を見せるべきか」もAIが判断する:

- card-carousel: レストラン・アクティビティ等の推薦カード

- map: 位置情報がある場合のインタラクティブ地図

- booking-widget: 予約可能な場合の予約UI

- photo-gallery: 画像がある場合のギャラリー

- qr-transfer: スマホへの情報転送QR（常時表示）

5.2 会話画面の原則

TVではチャットアプリ的な会話スクロールは行わない。最新の1ターンのみ表示し、前の会話を見たい場合は「さっき何話したっけ」と聞くか、戻るボタンで直前の応答に戻る。

結果が4件以上の場合は←→でページ送り、10件以上の場合はAIが「もっと絞りますか？」と提案する。

5.3 自然言語操作例（ゲスト側）

  ------------------------------------------------------------------------
  **ゲストの発話**                     **システム動作**
  ------------------------------------ -----------------------------------
  「Netflix見たい」                    直接Netflixアプリを起動

  「近くにおいしいラーメン屋ある？」   AIが推薦カードを表示 + 地図 + QR

  「スマホの画面をTVに映したい」       Chromecast接続手順を表示 + QR

  「テレビ消して」「おやすみ」         画面OFF or 深夜モード

  「なんか映画観たい」                 AIが好みをヒアリング → おすすめ →
                                       アプリ起動

  「このお店の情報スマホに送って」     QRコード表示 →
                                       スマホに地図・クーポン転送
  ------------------------------------------------------------------------

5.4 音声入力設計

マイクの常時ONはゲストの不信感を招くリスクが大きい。リモコンのマイクボタン押下方式（Push-to-Talk）を採用する。代替としてウェイクワード方式（「コンシェルジュ」）も検討。録音中は画面に🎙️アイコンを点灯し明示する。

6.2 ゲストログイン体験

ホテルTV最大の課題はゲストが自分のアカウントでログインした後、チェックアウト時にログアウトし忘れるリスク。QRログイン方式を推奨し、チェックアウト時に全アプリのログイン情報を自動消去する。

7\. スタッフ向け自然言語UIカスタマイズ

7.1 仕組み

ホテルスタッフは管理画面から自然言語でTV UIの設定を変更できる。Claude
APIがスタッフの要望を解釈し、テーマJSONを自動更新する。

操作フロー

1.  スタッフが自然言語で要望を入力

2.  AIがテーマJSONを生成

3.  リアルタイムプレビューを表示

4.  スタッフが承認 → 反映 / 「もう少し明るく」 → 再生成

8\. ゲストジャーニー設計

8.1 チェックインからチェックアウトまでの流れ

  -------------------------------------------------------------------------------------------------
  **フェーズ**            **TVの動作**                                      **データソース**

8.2 チェックアウト時の自動クリーンアップ

PMSのチェックアウトWebhookまたは予定時刻でトリガーされる:

- 全アプリのログインセッション消去

- ランチャーにリセット

- AI会話履歴消去

- ゲストのカスタム設定リセット

- 次のゲスト用ウェルカム画面準備

- 消去確認をPMSにログ記録（コンプライアンス）

9\. アクセシビリティと多言語対応

9.1 高齢者対応（シニアモード）

- フォントサイズ: 24px → 32pxに拡大

- ハイコントラスト表示

- アニメーション控えめ

- 全操作の音声ガイダンス

- 長押し誤操作防止

設定画面から切替、またはAIに「文字を大きくして」で有効化。

9.2 多言語対応

対応言語:
日本語、英語、中国語（簡体/繁体）、韓国語、タイ語、フランス語、スペイン語。

- 言語検出: PMS予約情報から国籍推定 →
  初期言語設定、または初回起動時に選択画面

- UIラベル: 事前翻訳済み（静的）

- AI応答: 何語でも対応（リアルタイム翻訳）

- 複数ゲスト: 直前の発話言語で応答（per-utterance切替）

9.3 子供モード

- アプリ制限: YouTube → YouTube Kidsに切替

- AI性格: フレンドリー・シンプル

- 設定変更にPIN必要（ペアレンタルロック）

9.4 視覚障害対応

Google
TV標準搭載のTalkBackを活用。フォーカス時の読み上げ対応を全UIコンポーネントに実装する。

11\. セキュリティとプライバシー

11.1 ゲスト間データ隔離

チェックアウト時に以下を全消去し、消去確認をPMSに監査ログとして記録:

- アプリログインセッション

- AI会話履歴

- キャスト履歴

- 検索履歴

- カスタム設定

- キャッシュされた個人データ

11.2 AI会話のプライバシー

- 会話は端末内のみ保存、退室時に消去

- 不適切な質問のコンテンツフィルタリング

- 部屋番号+名前はAIに渡すがAPI側にログを残さない

- APIログからPII自動マスク

11.3 デバイスセキュリティ

  -----------------------------------------------------------------------

  ----------------------------------- -----------------------------------
  Kiosk Mode                          ON（設定画面アクセス不可）

  USB                                 無効

  ADB                                 無効

  Factory Reset Protection            有効

  マイク                              Push-to-Talk（常時聴取禁止）

  録音インジケーター                  画面に🎙️点灯で明示

12\. オフライン耐性とエラーハンドリング

12.1 オフライン時の動作

ホテルのWi-Fiは不安定なことがある。AIが応答しなくなったらゲスト体験が崩壊するため、オフライン耐性を確保する。

ローカルキャッシュ対象

- 館内案内（全データ）

- 食事メニュー

- 温泉スケジュール

- FAQ 200問程度

- 周辺地図（静的）

- 緊急連絡先

AIフォールバック

オフライン時はFAQキーワードマッチングで回答。それでも無理な場合は「申し訳ございません。フロントにお電話ください。内線:
0番」を表示。

12.2 エッジケース

  ------------------------------------------------------------------------------------------------
  **ケース**                          **対応**
  ----------------------------------- ------------------------------------------------------------
  AIが答えられない質問                「確認いたします。フロントにおつなぎしますか？」
