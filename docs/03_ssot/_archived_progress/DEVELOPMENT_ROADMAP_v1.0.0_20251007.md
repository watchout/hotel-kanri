## 🎯 開発戦略

### 基本方針

#### 1. **SSOT First アプローチ**
```
SSOT作成 → 実装確認 → 差分修正 → 次のSSOT
```

**理由**:
- 実装済み機能が多いため、SSOT化を優先
- SSOTがないと実装の整合性が取れない
- 後から実装を修正するより、先にSSOTで統一

#### 2. **依存関係優先**
```
基盤SSOT → コア機能SSOT → 拡張機能SSOT
```

**理由**:
- 基盤SSOTが完成しないと、他のSSOTが書けない
- 認証・DB・マルチテナントは全機能の前提

#### 3. **実装済み優先**
```
実装済み機能のSSO化 → 未実装機能のSSO作成
```

**理由**:
- 実装済み機能は既に動いているため、SSOT化が速い
- 未実装機能は設計から必要なため、時間がかかる

#### 4. **管理画面優先、客室端末は後回し**
```
管理画面SSOT完成 → 客室端末SSOT作成
```

**理由**:
- 客室端末SSOTは管理画面SSOTに依存
- 管理画面が完成しないと、客室端末の仕様が決まらない

**発生確率**: 🟡 中（30%）  
**影響度**: 🔴 Critical  
**症状**: 
- 同じ概念が異なる名称で記載される
- データ型が統一されない
- 認証方式が一致しない

**対策**:
1. **Phase 0必須実行**: 既存SSOT全読み込み
2. **命名規則チェック**: 作成前に既存SSOTの命名を確認
3. **クロスリファレンス**: 関連SSOTへのリンク必須
4. **レビュー**: Izaが全SSOTをレビュー

**責任者**: Iza

**発生確率**: 🟡 中（40%）
